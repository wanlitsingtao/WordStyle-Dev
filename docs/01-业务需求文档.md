## 1. 项目概述

### 1.1 项目定位

WordStyle 是一个**文档样式转换工具**，主要面向投标人员，用于将招标文件快速转换为投标文件的应答文档。

### 1.2 核心价值

- **快速转换**：将源文档的样式批量转换为目标样式
- **批量处理**：支持同时处理多个文档
- **样式映射**：自动或手动映射源样式到目标样式
- **应答句插入**：支持在章节前后自动插入应答句
- **语气转换**：将客观描述转换为投标口吻

### 1.3 目标用户

- **主要用户**：投标人员、售前工程师
- **使用场景**：招投标项目中快速生成应答文档
- **用户特征**：需要处理大量文档，时间紧迫，要求转换结果格式规范

### 2.2 用户管理功能

#### 2.2.3 用户数据持久化

**持久化机制**：
- **用户初次访问时自动创建用户记录并写入数据库**
- **基于客户端设备标识生成唯一用户ID（12位MD5哈希字符串）**
- **所有用户操作实时记录到数据库**：
  - 文档转换历史（转换时间、文件名、段落数等）
  - 段落剩余额度（实时更新）
  - 用户活跃信息（最后登录时间、累计转换次数等）
- **管理后台可查看所有用户数据和操作历史**

**用户ID生成规则**：
```
# 1. 获取客户端设备标识（User-Agent）
user_agent = headers.get('User-Agent', 'unknown')

# 2. 生成设备指纹（基于User-Agent的MD5哈希）
device_fingerprint = md5(user_agent.encode())[:32]

# 3. 从数据库查询该设备指纹是否已有对应用户
query database for user where device_fingerprint = ?

# 4. 如果查询到用户，直接使用该用户ID
if user exists:
    return existing user.id

# 5. 如果未查询到，生成新用户ID并创建记录
else:
    # 5.1 生成用户ID（基于设备指纹）
    user_id = md5(f"wordstyle_device_{device_fingerprint}")[:12]
    
    # 5.2 创建新用户记录（包含device_fingerprint）
    create new user record in database with:
        - id: user_id
        - device_fingerprint: device_fingerprint
        - paragraphs_remaining: 10000
        - created_at: NOW()
        - last_login: NOW()
    
    # 5.3 返回新用户ID
    return new user_id
```

**跨会话持久化机制**：
- ✅ **真正的终端级唯一性**：同一终端（相同User-Agent）始终生成相同的用户ID
- ✅ **不依赖浏览器存储**：不使用localStorage、sessionStorage或cookie
- ✅ **不依赖会话状态**：即使Streamlit应用重启、session_state清空，用户ID仍保持不变
- ✅ **数据库驱动**：通过device_fingerprint字段在数据库中直接查询和关联用户
- ✅ **关机/重启/关闭浏览器无影响**：只要User-Agent不变，用户ID永远相同
- ✅ **每日额度自动重置**：查询到用户后，自动检查并重置当日免费额度为10,000段落

**API安全要求**（防止用户伪造身份）：
- 🔒 **禁止在URL参数中传递user_id**：防止用户通过修改URL获取其他用户数据
- 🔒 **使用设备指纹作为API查询标识**：所有API请求必须使用device_fingerprint而非user_id
- 🔒 **POST请求传递敏感参数**：设备指纹通过POST请求的JSON body传递，不在URL中暴露
- 🔒 **前端无法伪造设备指纹**：设备指纹基于User-Agent生成，存储在session_state，用户无法修改
- 🔒 **后端验证设备指纹有效性**：后端通过device_fingerprint查询数据库，返回对应的user_id和数据
- 🔒 **即使用户尝试修改也无效**：32位MD5哈希无法猜测，修改后查询不到用户数据

#### 2.2.2 额度管理

**免费额度**：
- 每日免费额度：10,000 段落
- 每日自动重置，不累计
- 新用户首次访问自动领取

**计费规则**：
- 按段落数扣费，具体费率由系统配置决定
- 用户余额和剩余额度实时记录在数据库中

#### 2.2.3 用户数据

**数据库表结构**（public.users 表，12个字段）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | VARCHAR(12) | 用户唯一标识，12位 MD5 哈希字符串 |
| `device_fingerprint` | VARCHAR(32) | 设备指纹，32位 MD5 哈希字符串（UNIQUE索引） |
| `username` | VARCHAR(50) | 用户名 |
| `style_mappings` | JSONB | 样式映射配置（JSON格式） |
| `balance` | FLOAT | 账户余额（元） |
| `paragraphs_remaining` | INTEGER | 剩余段落数 |
| `total_paragraphs_used` | INTEGER | 累计使用段落数 |
| `total_converted` | INTEGER | 累计转换文件数 |
| `is_active` | BOOLEAN | 是否激活 |
| `created_at` | TIMESTAMPTZ | 创建时间 |
| `last_login` | TIMESTAMPTZ | 最后登录时间 |
| `updated_at` | TIMESTAMPTZ | 更新时间 |

**存储内容**：
- 用户ID（12位字符串）
- **设备指纹（32位字符串，用于跨会话持久化）**
- 用户名
- 样式映射配置
- 账户余额（元）
- 剩余段落数
- 累计使用段落数
- 累计转换文件数
- 是否激活
- 创建时间
- 最后登录时间
- 更新时间

**注意事项**：
- ✅ 不使用微信登录，无需 `wechat_openid`、`wechat_nickname`、`wechat_avatar`
- ✅ 不使用邮箱登录，无需 `email`、`password_hash`
- ✅ 采用简单的用户ID机制，基于设备指纹（User-Agent）生成
- ✅ **支持真正的跨会话持久化**：通过数据库中的device_fingerprint字段实现
- ✅ **终端级唯一性**：同一终端始终对应同一用户ID，不受关机、重启、关闭浏览器影响

## 4. 业务流程

### 4.1 用户首次访问流程

```
1. 用户打开应用
   ↓
2. 系统检查是否存在用户ID
   ↓
3. 如无，生成新的用户ID（12位字符串）
   ↓
4. 检查用户数据是否存在
   ↓
5. 如无，创建新用户记录
   ↓
6. 检查是否是今日首次访问
   ↓
7. 如是，自动领取每日免费额度（10,000段落）
   ↓
8. 显示用户界面，展示剩余段落数
```

### 4.2 文档转换流程

```
1. 用户上传源文档（.docx）
   ↓
2. 用户上传模板文档（.docx）
   ↓
3. 系统分析源文档样式
   ↓
4. 系统分析模板文档样式
   ↓
5. 自动生成样式映射关系
   ↓
6. 用户可手动调整映射关系（可选）
   ↓
7. 用户配置转换选项（语气转换、应答句等）
   ↓
8. 用户点击"开始转换"
   ↓
9. 系统检查用户额度是否充足
   ↓
10. 如充足，执行转换
   ↓
11. 转换过程中显示进度条
   ↓
12. 转换完成后提供下载链接
   ↓
13. 扣除相应用户额度
   ↓
14. 更新用户统计数据
```

## 6. 限制与约束

### 6.1 技术限制

- 仅支持 .docx 格式（不支持 .doc）
- 单文件大小限制 50MB
- 复杂表格合并单元格可能存在样式丢失
- Visio 对象复制可能存在兼容性问题

### 6.2 业务限制

- MVP阶段采用手动充值方式
- 免费额度每日重置，不可累计
- 转换失败不退还额度（需人工处理）
- 临时文件保留7天后自动清理

### 6.3 平台限制

- Streamlit Cloud 免费套餐每月 750 小时
- Streamlit Cloud 不允许出站数据库连接
- Render 免费套餐服务会休眠（15分钟无请求）

### 6.4 安全约束

**用户身份识别安全要求**：

1. **禁止信任URL参数中的用户ID**
   - ❌ 严禁从 `st.query_params['uid']` 读取用户ID并直接使用
   - ❌ 严禁将用户ID写入URL参数（避免在地址栏暴露）
   - ✅ 必须基于设备指纹（IP + User-Agent）生成唯一标识
   
2. **设备指纹机制**
   - 使用客户端IP地址和User-Agent生成MD5哈希作为设备指纹
   - 通过本地映射文件（user_mapping.json）关联设备和用户ID
   - 同一设备始终使用相同的用户ID，防止身份伪造
   
3. **防刷机制**
   - 用户无法通过修改URL参数（如 `?uid=xxx`）来伪造身份
   - 用户无法通过清除浏览器缓存来领取新的免费额度
   - 每个设备只能领取一次新用户免费额度（10,000段落）
   
4. **已知攻击场景及防护**
   - **URL篡改攻击**：用户修改 `?uid=999999999999` → ✅ 已防护（忽略URL参数）
   - **清除缓存攻击**：用户清除localStorage/sessionStorage → ✅ 已防护（基于IP+UA）
   - **更换浏览器攻击**：用户使用不同浏览器 → ⚠️ 部分防护（UA不同会生成新ID）
   - **代理/VPN攻击**：用户切换IP地址 → ⚠️ 部分防护（IP不同会生成新ID）
   
5. **剩余风险与缓解**
   - **风险**：高级用户可通过更换IP或User-Agent生成新用户ID
   - **缓解措施**：
     - 对于普通用户，更换IP/UA的成本较高
     - 后端添加速率限制和异常检测（监控同一IP段频繁注册）
     - 考虑引入验证码机制（对可疑行为要求人机验证）
   
**历史漏洞记录**：
- 🚨 **严重漏洞**：用户可通过修改URL参数伪造身份，无限领取免费额度
- 📅 **发现时间**：2026年5月15日
- 🔧 **修复方案**：移除URL参数信任机制，改用设备指纹识别
- 📄 **详细文档**：参见 `安全漏洞修复_用户身份伪造.md`
