## 1. 项目概述

### 1.1 项目定位

WordStyle 是一个**文档样式转换工具**，主要面向投标人员，用于将招标文件快速转换为投标文件的应答文档。

### 1.2 核心价值

- **快速转换**：将源文档的样式批量转换为目标样式
- **批量处理**：支持同时处理多个文档
- **样式映射**：自动或手动映射源样式到目标样式
- **应答句插入**：支持在章节前后自动插入应答句
- **语气转换**：将客观描述转换为投标口吻

### 1.3 目标用户

- **主要用户**：投标人员、售前工程师
- **使用场景**：招投标项目中快速生成应答文档
- **用户特征**：需要处理大量文档，时间紧迫，要求转换结果格式规范

**功能描述**：
- 自动检测源文档样式
- 自动检测模板文档样式
- 支持手动调整样式映射关系

**映射规则**：
- Heading 1 → 标题 1
- Heading 2 → 标题 2
- Heading 3 → 标题 3
- Normal → 正文
- 支持用户自定义映射

#### 2.1.3 应答句插入

**功能描述**：
在Word文档中自动插入应答句，支持5种插入模式，满足不同投标场景需求。

**支持的插入模式**：

1. **章节前插入（before_heading）**
   - 在每个章节标题前插入应答句
   - 判断条件：如果标题后下一个元素不是标题，则在该标题前插入
   - 适用场景：需要在每个章节开始前明确表态

2. **章节后插入（after_heading）**
   - 在每个章节的正文最后一段后插入应答句
   - 判断条件：当前元素不是标题 + 下一个元素是标题
   - 特殊情况处理：
     - 全文档第一个标题前不插入（因为前面没有章节）
     - 两个标题之间不插入
     - 文章最后一段如果是正文，在其后插入
   - 适用场景：在章节内容结束后进行总结性应答

3. **章节后插入+章节原文副本（copy_chapter）**
   - 在章节末尾插入应答句
   - 复制从上一个标题后到当前位置的所有内容，插入到应答句之后
   - 形成结构：[章节内容] → [应答句] → [章节内容副本]
   - 适用场景：需要保留招标原文并进行应答对照

4. **逐段前应答（before_paragraph）**
   - 在非标题段落前插入应答句
   - 支持语义段落分组：将连续的短句、引号上下文、列表等语义相关段落分组
   - 在每个语义单元前插入一个应答句，避免过度插入
   - 适用场景：需要对每个具体需求点进行逐一应答

5. **逐段后应答（after_paragraph）**
   - 在非标题段落后插入应答句
   - 同样支持语义段落分组
   - 在每个语义单元后插入应答句
   - 适用场景：在阐述完每个需求后进行应答确认

**配置项**：
- **应答句文本**：自定义应答句内容（默认："应答：本投标人理解并满足要求。"）
- **应答句样式**：选择应答句的段落样式（从模板文档的样式列表中选择，默认：Normal）
- **插入模式**：从上述5种模式中选择一种

**技术实现要点**：
- 使用大纲级别（outline level）识别标题，而非仅依赖样式名称
- 支持处理表格、图片等非段落元素
- 采用XML级别的元素操作，确保格式完整性
- 内存中处理，减少文件I/O次数，提升性能
- 支持重试机制保存文档，提高稳定性

**用户界面**：
- 勾选"插入应答句"checkbox启用功能
- 启用后显示三个配置项：
  - 应答句文本输入框
  - 应答句样式下拉选择框（动态加载模板样式）
  - 插入模式下拉选择框（5种模式）
- 使用@st.fragment优化性能，避免不必要的重渲染

### 2.2 用户管理功能

#### 2.2.3 用户数据持久化

**持久化机制**：
- **用户初次访问时自动创建用户记录并写入数据库**
- **基于客户端设备标识生成唯一用户ID（12位MD5哈希字符串）**
- **所有用户操作实时记录到数据库**：
  - 文档转换历史（转换时间、文件名、段落数等）
  - 段落剩余额度（实时更新）
  - 用户活跃信息（最后登录时间、累计转换次数等）
- **管理后台可查看所有用户数据和操作历史**

**用户ID生成规则**：
```
# 1. 获取客户端设备标识（User-Agent）
user_agent = headers.get('User-Agent', 'unknown')

# 2. 生成设备指纹（基于User-Agent的MD5哈希）
device_fingerprint = md5(user_agent.encode())[:32]

# 3. 从数据库查询该设备指纹是否已有对应用户
query database for user where device_fingerprint = ?

# 4. 如果查询到用户，直接使用该用户ID
if user exists:
    return existing user.id

# 5. 如果未查询到，生成新用户ID并创建记录
else:
    # 5.1 生成用户ID（基于设备指纹）
    user_id = md5(f"wordstyle_device_{device_fingerprint}")[:12]
    
    # 5.2 创建新用户记录（包含device_fingerprint）
    create new user record in database with:
        - id: user_id
        - device_fingerprint: device_fingerprint
        - paragraphs_remaining: 10000
        - created_at: NOW()
        - last_login: NOW()
    
    # 5.3 返回新用户ID
    return new user_id
```

**跨会话持久化机制**：
- ✅ **真正的终端级唯一性**：同一终端（相同User-Agent）始终生成相同的用户ID
- ✅ **不依赖浏览器存储**：不使用localStorage、sessionStorage或cookie
- ✅ **不依赖会话状态**：即使Streamlit应用重启、session_state清空，用户ID仍保持不变
- ✅ **数据库驱动**：通过device_fingerprint字段在数据库中直接查询和关联用户
- ✅ **关机/重启/关闭浏览器无影响**：只要User-Agent不变，用户ID永远相同
- ✅ **每日额度自动重置**：查询到用户后，自动检查并重置当日免费额度为10,000段落

**API安全要求**（防止用户伪造身份）：
- 🔒 **禁止在URL参数中传递user_id**：防止用户通过修改URL获取其他用户数据
- 🔒 **使用设备指纹作为API查询标识**：所有API请求必须使用device_fingerprint而非user_id
- 🔒 **POST请求传递敏感参数**：设备指纹通过POST请求的JSON body传递，不在URL中暴露
- 🔒 **前端无法伪造设备指纹**：设备指纹基于User-Agent生成，存储在session_state，用户无法修改
- 🔒 **后端验证设备指纹有效性**：后端通过device_fingerprint查询数据库，返回对应的user_id和数据
- 🔒 **即使用户尝试修改也无效**：32位MD5哈希无法猜测，修改后查询不到用户数据

#### 2.2.2 额度管理

**免费额度**：
- 每日免费额度：10,000 段落
- 每日自动重置，不累计
- 新用户首次访问自动领取

**计费规则**：
- 按段落数扣费，具体费率由系统配置决定
- 用户余额和剩余额度实时记录在数据库中

#### 2.2.3 用户数据

**数据库表结构**（public.users 表，12个字段）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | VARCHAR(12) | 用户唯一标识，12位 MD5 哈希字符串 |
| `device_fingerprint` | VARCHAR(32) | 设备指纹，32位 MD5 哈希字符串（UNIQUE索引） |
| `username` | VARCHAR(50) | 用户名 |
| `style_mappings` | JSONB | 样式映射配置（JSON格式） |
| `balance` | FLOAT | 账户余额（元） |
| `paragraphs_remaining` | INTEGER | 剩余段落数 |
| `total_paragraphs_used` | INTEGER | 累计使用段落数 |
| `total_converted` | INTEGER | 累计转换文件数 |
| `is_active` | BOOLEAN | 是否激活 |
| `created_at` | TIMESTAMPTZ | 创建时间 |
| `last_login` | TIMESTAMPTZ | 最后登录时间 |
| `updated_at` | TIMESTAMPTZ | 更新时间 |

**存储内容**：
- 用户ID（12位字符串）
- **设备指纹（32位字符串，用于跨会话持久化）**
- 用户名
- 样式映射配置
- 账户余额（元）
- 剩余段落数
- 累计使用段落数
- 累计转换文件数
- 是否激活
- 创建时间
- 最后登录时间
- 更新时间

**注意事项**：
- ✅ 不使用微信登录，无需 `wechat_openid`、`wechat_nickname`、`wechat_avatar`
- ✅ 不使用邮箱登录，无需 `email`、`password_hash`
- ✅ 采用简单的用户ID机制，基于设备指纹（User-Agent）生成
- ✅ **支持真正的跨会话持久化**：通过数据库中的device_fingerprint字段实现
- ✅ **终端级唯一性**：同一终端始终对应同一用户ID，不受关机、重启、关闭浏览器影响

## 3. 非功能需求

### 3.1 性能需求

- **界面响应时间**：用户点击操作后 ≤ 3秒
- **文档转换时间**：根据文档大小，提供进度提示
- **并发支持**：支持多用户同时使用
- **文件上传限制**：最大 50MB

### 3.2 可用性需求

- **界面友好**：简洁直观的操作界面
- **错误提示**：清晰的错误信息和解决建议
- **进度反馈**：长时间操作提供进度条和状态提示
- **帮助文档**：提供使用说明和常见问题解答

### 3.3 安全性需求

- **用户隐私**：不收集个人敏感信息
- **数据安全**：用户数据存储在云端数据库
- **文件安全**：转换完成后自动清理临时文件
- **输入验证**：对用户上传的文件进行格式和大小验证

### 3.4 文件管理需求

#### 3.4.1 临时文件自动清理

**需求描述**：
- 用户上传的源文件（temp_source_*）在文档转换完成后立即删除
- 用户上传的模板文件（temp_template_*）在文档转换完成后立即删除
- 即使转换失败，也应尝试清理临时文件
- 清理操作不应影响转换结果文件的生成

**技术要求**：
- 使用try-finally确保清理逻辑执行
- 单个文件清理失败不影响其他文件
- 记录清理日志便于问题排查

#### 3.4.2 转换结果文件保留策略

**需求描述**：
- 转换后的结果文件为用户保留7天
- 超过7天的文件自动清理
- 保留期从文件生成时间开始计算

**存储位置**：
- 所有转换结果文件统一存储在 `conversion_results/` 目录
- 文件名格式：`result_{basename}_{timestamp}.docx`

#### 3.4.3 自动清理触发机制

**多种触发方式**：
1. **服务重启时清理**：服务启动时自动检查并清理过期文件
2. **每日定时清理**：每天零点（00:00）自动执行清理任务
3. **手动触发清理**：管理员可通过管理页面手动触发清理

**清理规则**：
- 清理超过7天的转换结果文件
- 清理遗留的临时文件（源文件、模板文件）
- 记录每次清理的文件数量和类型

#### 3.4.4 文件清理管理页面

**功能描述**：
- 提供管理后台的文件管理页面
- 以列表形式展示所有存储的文件
- 支持分页显示（当数据量较多时）

**列表信息**：
- 用户ID
- 文件名
- 文件类型（源文件/模板文件/转换结果）
- 文件生成时间
- 文件大小
- 文件状态（已过期/未过期）

**操作功能**：
- 单选：选择单个文件进行清理
- 多选：选择多个文件进行清理
- 全选：选择当前页所有文件
- 取消全选：取消所有选择
- 批量清理：清理选中的文件
- 清理过期文件：一键清理所有超过7天的文件

**特殊说明**：
- 虽然源文件和模板文件会自动清理，但保留此功能用于预防自动清理失败的情况
- 管理员可以手动清理任何类型的文件
- 清理操作需要二次确认，防止误删

### 3.5 可靠性需求

- **服务可用性**：7×24小时可用（依赖云平台）
- **数据持久化**：用户数据持久化存储
- **异常处理**：转换失败时保留错误信息供排查
- **自动恢复**：服务重启后自动清理过期数据

### 3.6 兼容性需求

- **浏览器兼容**：支持主流浏览器（Chrome、Firefox、Safari、Edge）
- **文档格式**：仅支持 .docx 格式

## 6. 限制与约束

### 6.1 技术限制

- 仅支持 .docx 格式（不支持 .doc）
- 单文件大小限制 50MB
- 复杂表格合并单元格可能存在样式丢失
- Visio 对象复制可能存在兼容性问题

### 6.2 业务限制

- MVP阶段采用手动充值方式
- 免费额度每日重置，不可累计
- 转换失败不退还额度（需人工处理）
- 临时文件保留7天后自动清理

### 6.3 平台限制

- Streamlit Cloud 免费套餐每月 750 小时
- Streamlit Cloud 不允许出站数据库连接
- Render 免费套餐服务会休眠（15分钟无请求）

### 6.4 安全约束

**用户身份识别安全要求**：

1. **禁止信任URL参数中的用户ID**
   - ❌ 严禁从 `st.query_params['uid']` 读取用户ID并直接使用
   - ❌ 严禁将用户ID写入URL参数（避免在地址栏暴露）
   - ✅ 必须基于设备指纹（IP + User-Agent）生成唯一标识
   
2. **设备指纹机制**
   - 使用客户端IP地址和User-Agent生成MD5哈希作为设备指纹
   - 通过本地映射文件（user_mapping.json）关联设备和用户ID
   - 同一设备始终使用相同的用户ID，防止身份伪造
   
3. **防刷机制**
   - 用户无法通过修改URL参数（如 `?uid=xxx`）来伪造身份
   - 用户无法通过清除浏览器缓存来领取新的免费额度
   - 每个设备只能领取一次新用户免费额度（10,000段落）
   
4. **已知攻击场景及防护**
   - **URL篡改攻击**：用户修改 `?uid=999999999999` → ✅ 已防护（忽略URL参数）
   - **清除缓存攻击**：用户清除localStorage/sessionStorage → ✅ 已防护（基于IP+UA）
   - **更换浏览器攻击**：用户使用不同浏览器 → ⚠️ 部分防护（UA不同会生成新ID）
   - **代理/VPN攻击**：用户切换IP地址 → ⚠️ 部分防护（IP不同会生成新ID）
   
5. **剩余风险与缓解**
   - **风险**：高级用户可通过更换IP或User-Agent生成新用户ID
   - **缓解措施**：
     - 对于普通用户，更换IP/UA的成本较高
     - 后端添加速率限制和异常检测（监控同一IP段频繁注册）
     - 考虑引入验证码机制（对可疑行为要求人机验证）
   
**历史漏洞记录**：
- 🚨 **严重漏洞**：用户可通过修改URL参数伪造身份，无限领取免费额度
- 📅 **发现时间**：2026年5月15日
- 🔧 **修复方案**：移除URL参数信任机制，改用设备指纹识别
- 📄 **详细文档**：参见 `安全漏洞修复_用户身份伪造.md`
